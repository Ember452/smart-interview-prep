# 面试流程展开版

本文件只保留 `SKILL.md` 中流程与出题策略的展开说明，用于维护和查阅；执行时以 `SKILL.md` 为准。

## 出题原则

### 优先级

1. **项目深挖**：优先围绕简历中真实项目追问设计决策、难点、取舍、结果
2. **技术栈核心概念**：围绕简历中写到的技术，优先问高频原理与工程实践
3. **行为面试**：根据经历考察沟通、推动、冲突处理、复盘能力
4. **系统设计**：仅当年限、岗位或项目深度明显支持时提高占比

### 禁止生成的问题

- 与简历完全无关的技术问题
- 纯背诵式八股，不结合场景或项目
- 冷门、论文级、面试低频问题
- 与已问问题高度重复的变体

### 出题扩展规则

出题时不限于简历显式写出的内容。基于简历中的技术栈和项目类型，面试官常会追问的关联方向也应纳入高优先级题池，例如：
- **同类技术对比**：简历写了 React，可追问"为什么选 React 而不是 Vue？当时做了哪些评估？"
- **规模扩大设计**：简历写了单机服务，可追问"如果日活涨 10 倍，你的架构需要怎么改？"
- **领域常见故障**：简历写了支付系统，可追问"支付回调丢单怎么设计补偿机制？"
- **团队协作延伸**：简历写了主导某个项目，可追问"你怎么推动跨团队技术决策？遇到分歧怎么处理？"
- **技术演进方向**：简历用了某个技术，可追问"这个技术最近几个大版本有什么变化？你怎么看它的演进方向？"

扩展边界：关联方向必须和简历中的技术栈、项目类型、业务领域有直接或合理关联，不得扩展到完全无关的冷门方向。扩展题目也应遵守追问上限和总题数边界。

## 一问一答原则（强制）

文本面试与真实语音面试不同，一次抛出多个问题会导致对话混乱、无法深入追问。面试全程必须遵守以下规则：

- **每次只问一个问题**。等候选人完整回复后，再基于其回答追问下一个问题。
- **严禁一次消息中抛出多个独立问题**（如"说说 Redis 的原理？还有消息队列怎么用？然后讲讲微服务拆分？"），这种多问多答模式在文字面试中必然导致混乱。
- **追问链按序执行**：一个问题 → 一个回答 → 分析回答 → 下一个追问，逐层深入。不要因为"想节省时间"而一次性把追问链全列出来。
- **场景设计题例外**：可以一次性给出业务背景、功能需求、性能约束等上下文信息（如"设计一个秒杀系统，日活 500 万，QPS 峰值 10 万，要求库存不超卖"），但不要在给出背景后同时要求回答多个独立分析维度。
- **阶段性说明例外**：在进入新阶段时（如从项目深挖切换到技术考察），可以简要告知接下来要考察的领域，但随后仍按一问一答执行。

违反此原则是文本面试中最常见的导致体验下降的原因，必须严格遵守。

## 两种模式流程

### 交互式模拟面试

1. 接收输入，优先处理聊天框粘贴的简历文本
2. 如用户给出文件，则尝试读取并提取文本；失败时引导改为文本粘贴
3. 解析简历，提取技术栈、项目、年限、角色、教育背景
4. 输出解析确认信息
5. 逐题提问，单题单轮推进
6. 根据回答质量决定追问或切换题目
7. `/report` 输出阶段报告，`/end` 输出完整报告

### 一键生成题库

1. 接收输入并解析简历
2. 一次性输出分类题库
3. 每题标注概率评级、考察点说明、追问方向提示
4. 不进入逐题问答，不启动追问机制

## 交互模式五步流程

### 第 0 步：解析与确认

- 提取技术栈、项目、工作经历、教育背景
- 参考 `templates/resume-confirm.md` 输出确认信息

### 第 1 步：开场热身

- 引导候选人介绍自己或最近做的项目
- 问题轻度，帮助进入状态

### 第 2 步：项目深挖

优先从以下方向出题：
- 架构设计
- 技术难点
- 性能优化
- 决策权衡
- 故障处理
- 复盘反思

### 第 3 步：系统设计（45 分钟以上独立安排，20/30 分钟融合到项目深挖）

以开放式命题开头，逐层追问展开：
- 给出业务背景、功能需求、性能约束（如"设计一个秒杀系统，日活 500 万，QPS 峰值 10 万，要求库存不超卖"）
- 追问链：数据模型 → 接口设计 → 架构组件 → 容量估算 → 瓶颈分析 → 扩展方案 → 数据一致性 → 容灾策略
- 常见场景分类：电商/交易（秒杀、支付、订单）、社交/内容（Feed 流、消息推送）、存储/数据（短链接、KV 存储）、基础设施（限流器、分布式 ID）

### 第 4 步：技术栈考察

遵循场景化提问原则：
- 不问纯定义
- 优先结合用户项目、真实业务量级、性能或故障场景发问

### 第 5 步：行为面试

- 考察学习能力、团队协作、解决冲突、项目推动、技术决策、跨部门协作等综合素质
- 根据候选人经历灵活选择提问方向

### 第 6 步：编码题（仅在 `include_coding == true` 时）

- 从 `reference/coding-challenges.md` 按技术栈选题
- 同步检索 LeetCode Hot 100 榜单补充算法题（优先中等难度）
- 代码 review 追问复杂度、边界、优化方向
- 卡住时按风格给提示

### 第 7 步：反问环节

- 必须主动邀请候选人反问："面试到这里基本结束，你有什么想问我的吗？"
- 根据面试官风格调整话术（`strict` 简洁直接，`gentle` 鼓励式引导）
- 回答 1-2 个问题后自然收尾，不强行延长

### 第 8 步：收尾

- 告知面试结束，生成完整报告
- `total_count >= 25` 或收到 `/end` 时触发收尾

## 追问策略

### 触发条件

| 用户回答特征 | 追问方向 |
|------------|---------|
| 回答过于概括 | 追问具体做法、数据、工具、结果 |
| 抛出技术名词 | 追问核心原理与项目中的实际使用 |
| 涉及方案选择 | 追问备选方案与 trade-off |
| 存在漏洞或模糊点 | 追问澄清和自洽性 |
| 回答明显优秀 | 追加更高难度场景题 |

### 禁止行为

- 同一话题超过 5 轮仍继续追问
- 用户明确表示不了解后仍强追同一方向
- 对无关技术点强行深挖
- 为了追问而追问，忽略整体面试节奏

## 简历关键词 → 追问链生成规则

解析简历时识别技术关键词，按以下规则预生成追问链：

### 追问链生成规则
1. 简历中出现的关键词必须至少追问 3 层
2. 每层追问必须基于上一层回答延伸（不是独立出新题）
3. 若回答不准确应追质疑，不跳过
4. 根据候选人实际回答质量动态控制终止深度：回答好就继续深挖，回答不上来就适可而止
5. 同一关键词可触发多个追问方向，但同一话题链内只追一个方向

### 通用高频关键词追问链

| 关键词 | 追问链 | 深度 |
|-------|--------|------|
| 缓存/Redis | 缓存策略选型 → 穿透/击穿/雪崩处理 → DB 一致性方案 → 分布式锁争议 → 淘汰策略选择 | 深入 |
| 消息队列 | 为什么用 → 不丢消息 → 顺序性 → 幂等性 → 堆积处理 → 架构演进 | 中等偏深 |
| 数据库/MySQL | 索引设计 → 慢查询分析 → 隔离级别 → MVCC → 分库分表考量 → 迁移容灾 | 中等偏深 |
| 并发/多线程 | 锁机制 → 线程池参数 → 并发工具对比 → 死锁排查 → 其他并发模型 | 深入 |
| 微服务/分布式 | 拆分原则 → 通信协议 → 治理（熔断/限流）→ 链路追踪 → 最终一致性 | 中等 |
| 容器/Docker/K8s | 镜像优化 → 编排策略 → 服务发现 → 资源限制 → 故障排查 | 中等 |
| LLM/AI/RAG | RAG 原理 → chunk 策略 → 召回指标 → 幻觉控制 → Agent/MCP 实践 | 中等偏深 |
| 性能优化 | 具体场景 → 量化数据 → 工具链 → 方法论 → 预防体系 | 深入 |

### 追问链执行规则
- 以用户回答实际内容为准，不机械按清单执行
- 用户回答已覆盖某层时，直接跳至下一层
- 用户在第 2 层回答充分时，不必强追到第 5 层

## 简历弱点预判与出题引导

解析简历后内部预判常见弱点，不向候选人展示，但用于指导出题方向：

| 简历信号 | 预判弱点 | 出题引导 |
|---------|---------|---------|
| 提到"负责"无具体技术细节 | 可能非技术主导 | 追问具体实现、代码贡献、架构决策 |
| 提到"优化"无量化数据 | 可能无实测 | 追问效果指标、测量方法、对比数据 |
| 技术名词堆砌但项目偏浅 | 可能只到使用层 | 追问核心原理、工程边界、故障处理 |
| 项目复杂度低（课程级） | 可能缺少生产经验 | 引入工程化问题：质量、协作、发布 |
| 技术栈窄（只一个框架） | 可能视野偏窄 | 追问"为什么选它？了解过其他方案吗？" |
| 年限与内容不匹配 | 可能经验虚标 | 通过追问验证深度，不直接质疑
