## 1. 数据分析

### 1.1 指标体系
- **北极星指标设计** — 平台核心价值量化的唯一关键指标（美团日交易笔数/抖音日活用户），反应业务健康度。不直接等于 KPI（如收入），但北极星指标提升通常带来商业指标增长
- **AB 测试统计方法** — 原假设 H0（两组无差异）。样本量：根据期望 MDE（最小检测效应）、显著性水平 α（通常 0.05）、统计功效 1-β（通常 80%）计算。检验：连续变量用 t 检验；比例类用 z 检验/卡方。结果：p < α 拒绝 H0，证明有统计显著性
- **辛普森悖论** — 分组数据趋势与总数据趋势相反。例：整体点击率上升，但男女分组均下降 → 因为男女样本比例结构变化。解法：分层分析/固定权重重新计算

### 1.2 漏斗分析
- **漏斗每一步转化率分析** — 注册→提交→支付：第一步转化低→入口问题（产品价值/引导）；中间环节低→流程体验/加载慢；最后一步低→价格/信任
- **归因模型** — 首次触点/末次触点/线性平均/时间衰减/位置衰减（首末多配）。选择取决于业务：品牌类看首次（哪个渠道拉新）、促销类看末次（哪个渠道促成转化）

### 1.3 数据质量
- **常见数据质量问题** — 缺失值（补全/删除/插值）、重复值（按业务键去重）、异常值（箱线图 1.5*IQR/Z-score 3σ 检测→业务验证后处理）、时间错乱（时间校准/序列修复）
- **数据血缘与治理** — 数据从源表 → ETL 转换 → 下游指标的全链路追踪（dbt/Atlas）；变更影响评估时顺着血缘定位所有下游消费方

## 2. 数据工程

### 2.1 ETL/ELT
- **ETL vs ELT** — ETL：抽取 → 转换 → 加载（数仓时代，先洗后存）；ELT：抽取 → 加载 → 转换（数据湖时代，先存原始数据后按需转换，灵活度高），适配云数仓（Snowflake/BigQuery 计算与存储分离）
- **批流一体** — 同一套 SQL 写一次代码同时服务批处理和实时流（Flink SQL/Kafka+ClickHouse）。Lambda 架构缺点(维护两套代码/两套存储) → Kappa 架构全流式

### 2.2 数据仓库分层
- **主流分层模型** — ODS 贴源层（原始数据）→ DWD 明细层（清洗/维度退化/数据质量）→ DWS 汇总层（轻度聚合/宽表）→ ADS 应用层（直接对接报表/API/BI）；维度表单独建模（DIM 层）
- **数据建模方法论** — 星型模型：事实表（度量 + 维度外键）+ 维度表（属性，扁平反范式化），查询简单性能好。雪花模型：维度表进一步规范化拆分，存储更省但 join 多查询慢；大部分场景星型更优

## 3. 机器学习

### 3.1 特征工程
- **特征编码** — 类别特征：One-Hot（低基数）、Label Encoding（高基数有序类别）、Target Encoding（高基数，用目标均值替换类别，但防过拟合需交叉验证）
- **特征选择方法** — Filter（方差/卡方/互信息/相关系数，与模型无关/计算快）、Wrapper（RFE 递归消除/前向搜索，需要反复训练模型/计算量较大）、Embedded（Lasso L1 正则化/树模型特征重要性，模型训练中同时完成选择）
- **数据标准化 vs 归一化** — 标准化 Z-score（x-μ）/σ：正态分布数据，适合 PCA/SVM/逻辑回归；归一化 Min-Max：（x-min）/（max-min）：保持原始分布比例，适合神经网络/基于距离的算法/KNN

### 3.2 模型评估
- **混淆矩阵指标** — Precision（预测正类的准确性：TP/（TP+FP），少误判关注）/ Recall（对真正样本的覆盖率：TP/（TP+FN），不遗漏关注）。F1=2*P*R/（P+R）调和平均。AUC-ROC：TPR vs FPR 曲线下面积，衡量整体排序能力
- **过拟合检测与解决** — 训练集表现远好于验证集/测试集。解决：增加训练数据、减少模型复杂度、正则化（L1/L2/Dropout）、早停（验证误差不再下降则停）、交叉验证、数据增强
- **离线 vs 线上指标不一致** — 原因：特征分布偏移（线上一段时间后与训练数据不同）、数据泄露（训练时用了线上拿不到的标签）、样本选择有偏（训练集不能代表线上请求分布）

### 3.3 线上线下一致性
- **特征一致性** — 离线训练/线上预测用同一套特征工程逻辑（Feathr/Feature Store）；特征日志实时采集并校验与训练集分布是否一致（Population Stability Index PSI）
- **推断延迟优化** — 模型量化（FP32→INT8，牺牲微小精度）/模型蒸馏（大模型教小模型）/Batch 预测合并/ONNX Runtime

## 4. LLM / AI 应用

### 4.1 RAG 检索增强
- **RAG 核心流程** — Query → Embedding → 向量召回 Top-K → 重排序（Reranker）→ LLM 生成。优势：注入外挂知识无需微调模型，幻觉率降低，支持知识动态更新
- **Chunk 策略** — 过大：语义不精确、超出 context window；过小：信息碎片化。平衡大小：512/1024 tokens。技巧：递归字符分割 → 语义分块（按段落/标题）→ 固定重叠率（chunk_overlap 10-20%）保上下文的连贯
- **召回优化** — 混合检索：关键词 BM25 + 语义向量融合。多轮对话场景需 Query 改写（抽取关键实体 + 合并对话历史）→ 多路召回（标题/摘要/全文分路）→ 粗排 → 精排（Cross-Encoder Reranker）
- **幻觉控制** — Grounded generation（生成必须基于检索结果，要求引用来源），置信度阈值（低置信度回复"我不知道"），引用校验后过滤矛盾答案

### 4.2 Agent
- **ReAct 推理范式** — Reason（思考该做什么）+ Act（调用工具/获取信息）交替循环。Thought → Action → Observation → Thought → ... → Final Answer。可处理多步推理和工具调用
- **工具调用（Function Calling）** — LLM 根据用户问题 + 工具描述 schema 决定调用哪个函数及参数，程序执行函数返回结果，LLM 解析结果输出最终答案。常见用途：查天气/搜文档/发邮件/操作数据库
- **Agent 循环风险** — 无终止条件会无限循环（设最大步数/超时）、工具返回错误导致循环中断（需重试/fallback 机制）、上下文窗口超额（生成摘要压缩前文）

### 4.3 Prompt 工程
- **Few-shot vs Zero-shot vs Chain-of-Thought** — Zero-shot：无示例直接提问；Few-shot：给 2-5 个输入输出示例引导格式；CoT：引导"让我们一步步思考"拆解推理步骤，对数学/逻辑/规划类任务效果提升显著
- **System Prompt 设计** — 角色设定 + 任务边界 + 输出格式约束 + 关键约束/禁止行为。越早越靠前权重越高；冲突约束中模型倾向遵守最近的指令

### 4.4 MCP 协议
- **MCP 三个角色** — Host（AI 客户端/IDE）、Client（协议客户端，一对一连接 Server）、Server（提供 Tools/Resources/Prompts 能力）。一个 Host 可连接多个 MCP Server（多能力聚合）
- **vs Function Calling** — MCP 标准协议接口（Server 方一次开发多客户端复用），Function Calling 耦合具体 LLM 平台。MCP 更像“通用 USB-C”，Function Calling 更像“专有充电口”

### 4.5 微调与模型评估
- **LoRA 原理** — 冻结原模型权重，仅训练低秩分解矩阵（A×B，秩 r=8-64），训练参数量减少 90%+。适用：特定领域知识注入、风格调整、指令遵循。局限：复杂新任务效果有限
- **RLHF 三阶段** — 1）监督微调（SFT：人工标注高质量输入输出对）；2）训练奖励模型（RM：从人类偏好中学习打分）；3）PPO 强化学习（用 RM 当裁判优化生成）。目标：让模型输出更符合人类偏好
- **LLM 评估基准** — MMLU（多任务语言理解，57 学科）；HumanEval（代码生成，164 题）；GSM8K（小学数学推理）；MT-Bench（对话质量，人工评估）；C-Eval（中文能力，52 科目）。评估陷阱：数据污染（训练集包含基准测试题）→ 需用新数据集交叉验证
