# 音视频与实时通信知识库

> 面试高频问题与答案要点。覆盖编解码/传输/直播/实时通信方向。

---

## 1. 编解码

- **H.264 vs H.265 vs VP9 vs AV1** — H.264 兼容性最好/最广泛支持/硬件编解码普遍；H.265（HEVC）同画质码率减半但专利费用高；VP9 开源免费 YouTube 主力、硬件支持渐广；AV1 下一代开源（同画质比 VP9 再低 30%）、编码慢但正在优化中
- **硬编 vs 软编** — 硬编（GPU/专用芯片）快速低功耗但画质调节余地小/参数可控差/兼容性受芯片限制；软编 CPU 占用高/功耗大但画质调优空间大/码率控制更精细。移动端场景优先硬编，PC/服务端转码用软编
- **GOP 与关键帧** — GOP（Group of Pictures）：一个 I 帧（完整画面/关键帧）+ 若干 P/B 帧（只存差异）。首屏延迟受 GOP 长度影响（等下一个 I 帧才能开始解码）。短 GOP=快速出画面但压缩效率下降；长 GOP=高压缩率但首屏/seek 延迟大

## 2. 传输协议

### 2.1 WebRTC
- **信令与媒体分离** — 信令通道（WebSocket/HTTP）只交换 SDP 元数据和 ICE Candidate，媒体数据走独立的 SRTP/SCTP 通道（加密+低延迟）。信令不经过媒体服务器，可自定义实现
- **ICE 连通性建立** — 收集所有候选地址：Host（本机 IP，直连最优）→ SRFLX（NAT 映射公网 IP，STUN 获取，多数情况可用）→ Relay（TURN 中转，高延迟/带宽开销，NAT 对称/防火墙严格时的兜底方案）
- **NAT 穿越与 TURN** — STUN 告诉客户端你的公网 IP 是什么、NAT 类型是什么；但对称 NAT 不通（每次目标不同端口不同映射）→ TURN 作为中继服务器，所有媒体流都经 TURN 转发，保证通但成本高
- **带宽估计（GCC）** — Google Congestion Control：基于丢包+延迟估计网络可用带宽；发送端根据估计带宽动态调整编码码率；接收端反馈 RTP 丢包/延迟到发送端。与 TCP 差异：不依赖 ACK/不无限重传（音视频宁丢不重传）

### 2.2 直播
- **RTMP vs HLS vs WebRTC-HTTP-FLV vs RTMP** — RTMP：推流→CDN，延迟 1-3s，基于 TCP 可靠但不快；HLS：切片文件（.ts 分片 + m3u8 索引），CDN 缓存友好、延迟 10-30s 以上、适合单播分发；WebRTC-HTTP-FLV：FLV 容器+HTTP 长连接、延迟 1-3s 比 RTMP 穿透防火墙更强
- **首帧/延迟优化** — 首帧：短 GOP（1-2s）/IDR 帧间隔缩到 1-2s 快速出画面/预加载播放器。延迟：降低 GOP/减少 B 帧/降低码率/接近源站边缘节点。常见直播延迟：传统 HLS 10-30s → LL-HLS 3-5s → HTTP-FLV 1-3s → WebRTC <1s

## 3. 音频处理链

- **3A 处理** — AEC（回声消除）：用自适应滤波器估计远端信号的回声路径并从近端信号中减去，关键参数 ERLE（回声消除比）目标 >30dB；ANS（噪声抑制）：谱减法/维纳滤波/VAD（语音活动检测）区分人声与背景噪；AGC（自动增益控制）：动态调整麦克风增益使输出音量稳定，避免削波失真
- **音频编解码选型** — Opus（全能：6kbps-512kbps、2.5ms-60ms 帧长、SILK 窄带语音 + CELT 宽带音乐自适应切换，WebRTC 默认）；AAC-LC（流媒体音乐标准）；G.711（传统电话 64kbps 无损 PCM，窄带 300-3400Hz）

## 4. 多人视频架构

- **SFU vs MCU vs Mesh** — Mesh（P2P 全连接，每人上行 N-1 路下行 N-1 路，带宽随人数指数增长，≤4 人可用）；MCU（服务端混流合成为一路，每人上下行各 1 路，服务端 CPU 重但客户端轻，适合大规模旁听）；SFU（服务端转发不混流，每人上行 1 路下行 N-1 路，服务端轻量可扩展，主流方案如 Jitsi/Mediasoup）
- **Simulcast 与 SVC** — Simulcast：发送端同时编码多路分辨率（720p/360p/180p），SFU 根据每个接收端带宽/屏幕大小选择性转发对应层；SVC（可伸缩视频编码）：单路编码分基础层+增强层，SFU 丢增强层不破坏基础层解码，带宽自适应更灵活但编解码复杂度高

## 5. 实时通信

- **长连接保活** — TCP 心跳（WebSocket ping/pong 应用层心跳 30-60s）、断线自动重连（指数退避：1s→2s→4s→...→max 60s）、重连后状态同步（同步最新消息序号/恢复会话）。避免依赖系统 TCP Keep-Alive（默认 2h 太长且 NAT 可能提前淘汰）
- **消息时序与去重** — 发送端带单调递增序号，接收端按序号排序；乱序消息入暂存缓冲，超时后丢弃（实时通信宁丢不延迟）。去重：服务端维护已处理消息 ID Set（TTL 防内存泄漏），客户端幂等处理
 **万人房间推送方案** — 分房间分桶（同一房间 200-500 人一桶，桶内全量推送 + 桶间聚合）。信令/状态类消息广播；高优消息（礼物/红包）走单独队列；离线降级（拉模式获取历史消息不丢）+ 在线推模式实时

## 6. QoS/QoE 监控指标

- **核心指标** — 端到端延迟（采集→编码→网络→解码→渲染全链路，RTC 目标 <300ms）、卡顿率（每 100 分钟卡顿总时长，<1% 为优秀）、首帧时间（从加入房间到首帧渲染 <500ms）、音画同步偏差（<100ms 人耳不可感知）
- **弱网对抗策略** — 丢包：FEC（前向纠错，冗余数据包，丢包 <10% 可恢复）+ NACK（丢包重传请求，RTT 低时有效）+ PLC（丢包隐藏，用前一帧外推）；抖动：Jitter Buffer（缓冲队列平滑延迟抖动，典型 100-200ms）；带宽受限：SVC 降级 + Simulcast 切低层 + 降低帧率/分辨率
