## 1. 数据库

### 1.1 索引
- **B+Tree 索引结构** — 所有数据存于叶子节点（非叶节点只存键 + 指针），叶子节点之间双向链表连接，全表扫描无需多次寻址；相比 B-Tree 磁盘 I/O 更少（非叶节点不存数据，一个节点可容纳更多键），范围查询高效
- **聚簇索引 vs 非聚簇索引** — 聚簇索引叶子节点即数据行（每表一个，默认主键），非聚簇索引叶子存主键值，需回表
- **索引失效场景** — 条件前加函数/运算（WHERE YEAR(date)=2024）、前置模糊（LIKE '%xx'）、类型隐式转换（varchar 字段用整型查）、联合索引不满足最左前缀、负向条件（!=、NOT IN）通常不走索引但或走全索引扫描
- **覆盖索引** — 查询所有需要的列都在同索引中，无需回表，直接返回。判断：EXPLAIN 中 Extra 有一列 `Using index`。设计方式：把 SELECT 的列加入联合索引
- **联合索引最左前缀** — 联合索引 (a,b,c) 等效创建 (a)、(a,b)、(a,b,c) 三个索引。查询条件必须从最左列开始才能用到索引

### 1.2 事务与隔离级别
- **四大隔离级别** — READ UNCOMMITTED（脏读）/ READ COMMITTED（不可重复读，Oracle 默认）/ REPEATABLE READ（幻读，MySQL InnoDB 默认，通过 MVCC + Next-Key Lock 解决）/ SERIALIZABLE（串行化）
- **MVCC 原理** — 每行数据有隐藏列：trx_id（最近修改事务 ID）+ roll_pointer（undo log 回滚指针）。ReadView 判断某版本是否对当前事务可见：已提交 + 在 ReadView 生成前提交 → 可见。REPEATABLE READ 级别下 ReadView 在事务开始时生成；READ COMMITTED 下每次 SELECT 生成新 ReadView
- **InnoDB 如何解决幻读** — MVCC 快照读（普通 SELECT）避免幻读；当前读（SELECT...FOR UPDATE）用 Next-Key Lock（行锁 + 间隙锁）锁住不存在的行插入

### 1.3 慢查询治理
- **慢查询排查流程** — EXPLAIN 看 type（ALL/index/range→需优化）→ 看 key/rows/Extra（filesort→排序无索引；temporary→临时表）→ 看是否有隐式转换 → 确认是否缺覆盖索引 → 考虑分库分表/读写分离
- **SQL 优化常见手段** — 避免 SELECT *（网络+回表代价）、分页优化（子查询定位+关联 替代大 offset LIMIT）、IN 值太多换 JOIN、批量操作控制事务大小

### 1.4 分库分表
- **何时需要分库分表** — 单表数据量 >2000W 或 >10GB 性能开始下降（索引深度增加）；单库 QPS>1000 或连接数压力大
- **分片策略** — 水平分片（range/hash/list）：range 查询友好但热点集中（最新数据总在同一库）、hash 均匀但跨分片查询困难；垂直分片按业务域拆（用户库/订单库）
- **分库分表后的问题** — 跨库 JOIN 要拆为多次单表查询应用层合并；分布式事务用 TCC/Saga；全局 ID 用雪花算法/号段模式；扩容需数据迁移

## 2. 缓存

### 2.1 Redis 底层数据结构
- **5 种基础类型 + 底层实现** — String（SDS 简单动态字符串）、List（双向链表/quicklist）、Hash（压缩列表/哈希表）、Set（intset/哈希表）、ZSet（压缩列表/跳表+哈希表）
- **Redis 为什么快** — 单线程内存操作（无锁/无上下文切换）、IO 多路复用（epoll 监听多个连接并单线程处理）、数据结构简单高效、RESP 协议解析轻量
- **Redis 6.0+ 多线程** — 仅网络 I/O 多线程（读写 socket/解析请求），命令执行仍单线程。解压/压缩、大键删除等后台线程异步处理

### 2.2 缓存问题
- **缓存穿透** — 查询不存在数据 → 请求穿透缓存 → 打到 DB。方案：布隆过滤器提前判断 key 是否存在；空值缓存短时间（但需定期清理避免内存浪费）
- **缓存击穿** — 热点 key 过期 → 大量请求同时打 DB。方案：互斥锁（同一时间只一个线程查 DB 写缓存）；逻辑过期（缓存放永不过期的值 + 后台续期）
- **缓存雪崩** — 大量 key 同时过期 → 数据库压力陡增。方案：过期时间加随机抖动；多级缓存（本地 + Redis）；服务降级/限流

### 2.3 缓存一致性
- **为什么不是强一致** — 更新 DB 成功 → 删缓存失败，读到旧数据。CAP 中可用性优先，缓存本质是最终一致
- **Cache Aside 模式** — 读：先读缓存，未命中则读 DB → 写缓存。写：先更新 DB → 再删除缓存（不是更新缓存，避免并发写顺序问题）
- **延迟双删** — 先删缓存 → 更新 DB → 延迟 N 毫秒再删缓存，防止并发读写造成的短暂脏数据（适用于读写分离主从延迟场景）

### 2.4 分布式锁
- **Redis 分布式锁正确姿势** — SETNX + EXPIRE（两条指令不原子 → 死锁风险）→ 用 `SET key value NX EX seconds` 一步到位 → 解锁用 Lua 脚本校验 value 再 DEL（避免误删别人的锁）→ Redisson 看门狗监控续期
- **Redlock 争议** — 多 Redis 节点多数投票加锁。Martin Kleppmann 指出 GC 暂停/时钟跳跃可能导致锁失效；Antirez 回应 fencing token 可解决

## 3. 分布式系统

### 3.1 CAP 与一致性
- **CAP 实践理解** — 三个无法同时满足。P（分区容错）必须保证（网络不可靠，跨机房必然有延迟）；通常在 C 和 A 之间权衡：核心数据选一致（CP-ZK/etcd），边缘数据/高并发选可用（AP-Eureka/Nacos 注册中心）
- **Paxos / Raft 共识** — Raft 将一致性拆为 Leader 选举 + 日志复制 + 安全性保证，更易理解。Leader 接收所有客户端请求，向 Follower 复制日志，过半确认后提交

### 3.2 分布式事务
- **刚性事务 vs 柔性事务** — 刚性：XA 二阶段提交，强一致但性能差、锁定时间长；柔性：TCC（Try-Confirm-Cancel 两阶段业务补偿）、Saga（正向调用链失败时反向补偿）、可靠消息最终一致（本地消息表/事务消息）
- **TCC 要点** — Try 预留资源 → Confirm 提交资源/Cancel 释放资源。Cancel 操作必须幂等；空回滚处理（Try 未执行但 Cancel 被调用）

### 3.3 分布式 ID
- **雪花算法设计** — 64 位：1b 符号位 + 41b 时间戳（69 年）+ 10b 机器ID + 12b 序列号（每毫秒 4096）。问题：时钟回拨（等待/记录最大时间戳/借用未来序列号）

## 4. 消息队列

- **消息不丢失三阶段** — 生产端：同步发送 + 确认回调 + 重试（confirm/ack）→ 存储端：多副本同步刷盘 + min.insync.replicas 确认 → 消费端：先处理再提交 offset
- **消息顺序性** — 全局有序：Kafka 单分区，生产端单线程发送，消费端单线程消费。局部有序（同 ID 进入同分区）：生产端指定 key hash 分区，消费端分区内有序
- **消息幂等** — 消费端做幂等：数据库唯一键去重（插入 insert ignore）；Redis setnx 标记已处理；业务状态机（如已支付不再重复处理）
- **消息堆积** — 原因：消费跟不上生产。应对：紧急扩容消费者实例（不超过分区数）、批量消费优化；长期：增加分区、消息过期策略、监控告警

## 5. 接口设计

- **RESTful 设计原则** — 资源导向（URL 为名词复数）、HTTP 方法语义对应 CRUD、无状态、版本控制（URL/v1/- 或 Accept header）、HATEOAS 超链接驱动的可发现性
- **GraphQL vs REST** — GraphQL 客户端精确指定字段（避免 over/under fetch）、单入口、强类型 schema；代价：复杂度后移服务端、缓存难（POST 方式）、深度嵌套 N+1 问题（DataLoader 批处理解决）
- **gRPC 特点** — Protobuf 二进制序列化（体积小/解析快/强类型）、HTTP/2 多路复用 + 双向流、IDL 定义接口自动生成多语言代码；缺点：浏览器直调困难（需 gRPC-Web）、人类不可读、调试门槛高
- **接口向后兼容原则** — 只增加字段不删除、不修改字段含义/类型、废弃字段用标记而非删除、新版本独立路径或内容协商（Accept header）
