## 1. 计算机网络

### 1.1 传输协议
- **TCP 三次握手与四次挥手** — 三次握手：SYN（Client→Server）→SYN+ACK（Server→Client）→ACK（Client→Server）。防旧连接请求乱序到达。两次握手：Server 确认 Client 有发送能力，但 Client 不能确认 Server 有发送能力也不确认 Server 接收能力。四次挥手：FIN（主动方→被动方）→ACK（被动方→主动方）→FIN（被动方→主动方）→ACK（主动方→被动方）。TCP 全双工，双方独立关闭发送通道，被动方收到 FIN 后可能还有数据需发，必须分两次。TIME_WAIT 2MSL 等待（主动关闭方保证最后的 ACK 能到达+让旧连接报文彻底消失）
- **TCP 拥塞控制四阶段** — 慢启动（CWND 指数增长，每 RTT 翻倍→到 ssthresh 阈值切换）。拥塞避免（CWND 线性增长，每 RTT 加 1）。快速重传（收到 3 个重复 ACK 不等到 RTO 超时直接重传丢失包）。快速恢复（收到重复 ACK 不减 CWND 到 1 而是降到 ssthresh，保持在拥塞避免阶段不进入慢启动，提升恢复速度）
- **QUIC / HTTP/3 vs TCP/HTTP/2** — QUIC 基于 UDP 空间上实现了类似 TCP 的可靠 + TLS 内置（永远加密，无需额外握手回合）+ 无队头阻塞（TCP 一个流丢包阻塞同连接所有流；QUIC 各流独立重传不互阻塞）+ 连接迁移（无线网络切换不重建连接，通过 Connection ID 识别会话）。TCP 在内核态不可改 -> QUIC 在用户态可快速迭代

### 1.2 应用协议
- **HTTP/1.1 vs HTTP/2 vs HTTP/3** — HTTP/1.1：Keep-Alive 复用连接但有队头阻塞（前一个请求没回来后面要等）、多个 TCP 连接才能并发（6 个浏览器限制）。HTTP/2：多路复用（一个 TCP 连接上并发多 Stream，各自独立 Frame 不相互阻塞，但有 TCP 层队头阻塞）。Header 压缩（HPACK 静态表+动态表去重，不再重复发相同 Header）。Server Push（服务器可主动推送 CSS/JS 到客户端）。HTTP/3：QUIC 解决 TCP 队头阻塞+更快的握手建立
- **HTTPS TLS 1.3 握手** — TLS 1.2 握手（4 次往返）：Client Hello → Server Hello+Certificate → Client Key Exchange → Server Finished →共计 2-RTT。TLS 1.3（1-RTT）：Client Hello（带 Key Share 预判密钥）→ Server Hello+Certificate+证书验证+Finished→ 一次性完成，共 1-RTT（首次）。再次访问时 0-RTT（预共享密钥 PSK，在 Hello 消息中直接加密应用数据，但无前向安全性）
- **WebSocket vs HTTP** — HTTP 请求-响应单次/客户端主动发起。WebSocket（HTTP Upgrade:websocket 从 HTTP 升级到长连接）全双工实时通信，服务端可主动推送消息到客户端。适合聊天/实时游戏/协作编辑/实时大盘

### 1.3 DNS
- **DNS 解析完整链路** — 浏览器缓存 → OS hosts file → 本地 DNS 服务 → 根域名服务器 → 顶级域名服务器（.com）→ 权威 DNS → 记录 IP → 沿途缓存。CDN （DNS 智能解析按用户 IP 地理位置返回最近的 CDN 节点 IP，就近加速）
- **DNS 污染与劫持** — DNS 污染：中间节点伪造 UDP DNS 响应（无加密无认证，伪造响应比真正的响应先到达就能骗过客户端）。DNS 劫持：运营商/路由器控制 DNS 服务器，按自己的意愿返回结果（强制广告/重定向）。对抗：DoH（DNS over HTTPS，DNS 查询走加密 HTTPS 连接，运营商不可见/不可篡改）/DoT（DNS over TLS，同加密但独立端口，可被防火墙拦截）

## 2. 数据结构与算法

- **HashMap 与字典实现** — 数组+链表/红黑树（JDK 1.8+）。Hash 函数计算 key 的 bucket index→ O(1)。冲突处理：链表（同 bucket index 的多个 entry 拉链）→高冲突后链表过长退化为 O(n) →链表长度超过 8+数组容量>=64 时转为红黑树 O(log n)。扩容 rehash（容量×2 负载因子 0.75，数据迁移到新容量 rehash）
- **LRU 实现原理** — 哈希表+双向链表。get：查表 O(1)、移动到链表头部标记最近使用。put：存在更新值移头→不存在链表尾淘汰最久未使用→插入新节点头部 O(1)。扩展进化：LRU-K（记录访问时间历史，K 次访问统计计算淘汰值，更准确辨识热点）
- **常见排序算法复杂度场景** — 快排平均 O(n log n) 几乎最快，基准 pivot 随机选择防最坏 O(n²) 有序退化。归并 O(n log n) 稳定/O(n) 额外空间/适合外部分拣大数据（磁盘）。堆排 O(n log n) 不稳定/原地/适合找 Top K 问题。不稳定选择要考虑稳定性要求+数据规模+是否随机分布

## 3. 设计模式

- **单例模式安全实现** — 饿汉式：类加载时创建（线程安全但可能浪费且启动慢/反序列化破坏单例）。懒汉式双重检查锁定：volatile 禁止指令重排 + synchronized 锁类对象 + 二层判空，99% 调用不走同步块。静态内部类持有实例、JVM 保证唯一/延迟加载无同步开销/最优解
- **策略模式 vs 工厂模式区别** — 工厂模式创建不同产品（根据输入参数返回不同类实例，关注对象创建）；策略模式在不同算法/行为间切换（算法封装为独立类+可运行时替换实现，关注行为选择）。实际中常搭配：工厂创建策略实例
- **观察者模式双模型** — 推模型（被观察者主动推送所有数据到观察者，易不匹配观察者需要的具体数据造成数据过载）；拉模型（只通知观察者有更新，由观察者按需返回拉取部分数据，灵活但观察者与被观察者耦合更高）。典型实现：发布订阅模型解耦（EventBus/消息中间件/Broker），三方事件通道，双方不相互直接引用

## 4. DevOps / CI/CD

- **分支策略对比** — GitFlow（主分支+develop+feature/release/hotfix，严格但沉重/迭代速率低的传统软件）；GitHub Flow（仅 main+feature 分支，PR review 合并→立即可部署，简单/快速，适合持续部署 Web 服务）；Trunk Based（只一个主干，所有开发者向主干提交，极短分支 < 1 天+功能标志保护未完成功能，极致速率+高工程纪律要求）
- **CI/CD 流水线设计** — CI（代码提交→自动构建→自动化测试→打包产物，每个 commit 都通过，问题第一时间暴露不积累）。CD（CI 后自动部署到各环境，Staging 全真环境与生产一致→验证通过→生产 deploy，全程自动无需人工操作但保留手动审批按钮）。门禁失败阻断继续向下游推进
- **Dockerfile 最佳实践** — 多阶段构建分离构建依赖与运行依赖→ 产物镜像里没有 SDK/中间编译产物 体积缩小 90%。分层利用缓存（从最少变动的指令排列开始，如先 COPY package.json 再 npm install 再 COPY 源码，变更源码时不用 reinstall 依赖）。不推荐 latest tag（指向不明/无法回滚到确切版本），Tag 用 commit hash/时间戳 可追溯

## 5. 操作系统基础

- **进程 vs 线程 vs 协程** — 进程：独立地址空间，进程间通信需 IPC（管道/共享内存/Socket），资源开销大但隔离性强；线程：共享进程地址空间，切换开销小于进程，需同步原语保护共享数据；协程：用户态轻量级线程，切换无内核开销，适合 IO 密集型高并发（Go goroutine / Python asyncio / Kotlin coroutines）
- **虚拟内存与分页** — 每个进程拥有独立虚拟地址空间（32 位 4GB/64 位巨大），通过页表映射到物理内存。缺页中断（Page Fault）：访问未映射页 → 内核从磁盘加载页面到物理内存 → 更新页表。Swap 空间：物理内存不足时将不活跃页面交换到磁盘，触发时延迟高
- **进程同步原语** — 互斥锁 Mutex（同一时间只一个线程进入临界区）；信号量 Semaphore（计数锁，允许 N 个线程同时进入）；条件变量（等待某个条件成立时阻塞，收到信号后唤醒）；读写锁 RWMutex（读读不互斥，读写/写写互斥，适合读多写少场景）

| 多线程+锁 | Java/C++ | 共享内存，显式锁，调试复杂 | CPU 密集型计算 |
