## 1. LLM 底层原理

### 1.1 Token 与上下文窗口

- **Token 是什么？** — 文本处理的最小单位。英文：1 Token ≈ 4 个字符或 0.75 个单词；中文：1 Token ≈ 1-2 个汉字。GPT 系列使用 BPE（字节对编码）分词器
- **上下文窗口（Context Window）？** — 模型一次能处理的最大 Token 数量，包括输入和输出。超过会截断
- **上下文窗口满了怎么办？** — 滑动窗口截断历史；摘要脱水；分层记忆；RAG 检索相关片段替代完整历史
- **为什么中文比英文消耗更多 Token？** — BPE 对不同语言效率不同，汉字在训练语料中复现率低，分词粒度更细

### 1.2 Temperature 与采样参数

- **Temperature 的作用？** — 控制输出随机性。0 = 确定性（代码生成）；0.3-0.7 = 平衡（对话）；1.0+ = 创造性（头脑风暴）
- **Top-p 和 Top-k 的区别？** — Top-k：从概率最高的 k 个词中采样；Top-p：从累计概率达到 p 的最小词集中采样，更常用
- **结构化输出为什么还要指定 Temperature？** — 即使指定 JSON 格式，模型仍需在多个合法选项中做选择

### 1.3 模型能力与局限性

- **幻觉（Hallucination）为什么产生？** — LLM 是语言模型而非知识数据库；训练数据有错误；缺乏实时验证机制；注意力机制在不存在的模式中"找到"联系
- **如何缓解幻觉？** — RAG 检索增强；提示词约束（要求引用原文）；CoT 链式推理；工具调用验证；事实核查层
- **LLM 的"失忆"问题？** — 上下文长度限制导致早期信息被截断；位置编码衰减导致"Lost in the Middle"

## 2. AI Agent

### 2.1 Agent 核心概念

- **Agent 是什么？** — Agent = LLM + Planning + Memory + Tools。能感知环境、决策并执行动作的自主系统
- **Agent Loop 是什么？** — 本质是 while 循环：LLM 推理 → 工具调用 → 上下文更新 → 重复，直至任务终止
- **Agent vs Workflow vs 传统编程的区别？** — 传统编程和 Workflow 是人做决策，Agent 是 AI 做决策。灵活性、门槛和维护成本不同
- **Agent 的核心挑战？** — 上下文窗口限制；幻觉；Token 经济性；工具安全边界（Prompt Injection）；规划能力上限

### 2.2 ReAct 与推理范式

- **ReAct 是什么？** — Reasoning + Acting，"思考→行动→观察"循环完成任务，减少幻觉、提升可解释性
- **ReAct vs Plan-and-Execute？** — ReAct 动态逐步调整；Plan-and-Execute 先制定全局计划再执行
- **Reflection 反思模式？** — 赋予 Agent 自我纠错能力：Reflexion（失败后反思）、Self-Refine（输出后批评改进）、CRITIC（外部工具验证）
- **Multi-Agent 系统？** — 多个独立 Agent 协作。主流架构：Orchestrator-Subagent（编排者-子代理）；Peer-to-Peer（对等协作）

### 2.3 Context Engineering

- **Context Engineering 是什么？** — 为 LLM 构建高信噪比信息输入环境的技术，在有限 Token 窗口内提供精准决策依据
- **System Prompt 怎么设计？** — 高度结构化 Markdown：角色设定、核心目标、严格约束、标准执行流、输出格式
- **上下文太长怎么优化？** — 早期对话压缩为摘要；RAG 检索资料二次裁切；系统约束和核心工具描述绝对保护；Context Caching 降低并发成本

## 3. RAG 检索增强生成

### 3.1 RAG 核心原理

- **RAG 是什么？** — Retrieval-Augmented Generation，检索增强生成。解决 LLM 知识过时、幻觉、截止日期问题
- **RAG 完整流程？** — 索引：文档加载 → 文本分割 → 向量化 → 存入向量数据库；查询：问题向量化 → Top-K 检索 → 上下文组装 → LLM 生成
- **RAG vs Fine-tuning？** — RAG：动态知识、频繁更新、需要可解释性；Fine-tuning：固定模式、风格学习、复杂推理调整

### 3.2 文档分块策略

- **Chunk Size 怎么选？** — 固定分块（512-1024 Token）；句段落分块（尊重语义）；递归字符分块；语义分块（基于 Embedding 自动判断）
- **重叠分块（Overlap）的作用？** — 在相邻 Chunk 之间保留重叠内容，防止语义在分割处断裂。常用 10-20%

### 3.3 向量检索与索引

- **常见向量索引算法？** — FLAT（暴力检索）、IVF（倒排索引）、HNSW（分层导航小世界图，高召回低延迟）、PQ（乘积量化压缩）
- **余弦相似度 vs 点积 vs 欧氏距离？** — 余弦：方向相似，不受长度影响；点积：综合方向和幅度；欧氏：绝对距离
- **向量数据库怎么选？** — Milvus（大规模高性能）、Pinecone（托管方便）、Chroma（轻量本地）、Qdrant（支持过滤）、Weaviate（混合搜索）

### 3.4 召回质量优化

- **召回率低怎么优化？** — 检查向量模型匹配度；调整 Chunk Size；增加 Overlap；混合检索（向量+关键词）；重排序（Re-ranker）
- **Re-ranker 的作用？** — 两阶段：粗召回（向量 Top-K 大）→ 精排（Cross-Encoder Top-N 小）
- **混合检索是什么？** — 向量检索（语义相似）+ 关键词检索（BM25），用 RRF 融合多路结果

### 3.5 RAG 优化验证追问链（重要）

> 当候选人提到 RAG 优化时，必须连续追问验证，避免"堆砌技术"而不知效果。

**追问链（必须连续追问）：**
1. **你怎么验证的？** — 期望：有人工评估数据集、与 baseline 对比。"体感提升了"→ 继续追问
2. **量化指标是什么？提升了多少%？** — 期望：Recall@K、MRR、RAGAS 等具体数字。无法给出 → 继续追问
3. **消融实验有吗？** — 期望：逐步去掉每个优化项看效果变化
4. **知识库规模多大？** — 建议 500+ 文档才能说明优化有效。只有 10-20 条 → 指出规模太小

**常见错误回答模式（面试官需质疑）：**
- "体感上提升了" → 体感不是度量标准
- "用了社区常用的方法" → 常用不代表有效
- "知识库有 14 篇文章" → 数据量太小，优化无意义

## 4. MCP 协议与工具调用

### 4.1 Function Calling

- **Function Calling 是什么？** — 允许 LLM 调用外部函数。模型输出结构化函数调用请求，应用解析并执行。底层依赖 JSON Schema 描述函数签名
- **好的工具描述怎么写？** — 明确"何时该调用"和"何时不该调用"；参数包含格式要求和示例值
- **Function Calling 的局限性？** — 依赖模型支持；参数填充可能出错；多函数顺序依赖难处理

### 4.2 MCP 协议

- **MCP 是什么？** — Model Context Protocol，Anthropic 推出的开放协议。统一 AI 应用与外部工具的通信，"AI 领域的 USB-C"
- **MCP 的三大核心原语？** — Tools（可执行函数）、Resources（只读数据）、Prompts（可复用提示词模板）
- **MCP 和 Function Calling 的关系？** — JSON Schema 是底层数据格式标准；MCP 是其之上的通信协议层。两者正交，可叠加使用

### 4.3 工具安全

- **Prompt Injection 是什么？** — 攻击者通过外部输入试图覆盖 Agent 系统指令
- **防护手段？** — 沙箱隔离、权限最小化；Prompt 隔离；高危操作人机审批

## 5. AI 编程实战

### 5.1 AI 编程工具使用

- **主流工具？** — GitHub Copilot、Cursor、Windsurf、Trae、Claude Code 等
- **核心原理？** — 上下文感知补全（当前文件+相邻文件+注释）；自然语言转代码；多模态理解
- **好的 AI 编程 Prompt？** — 明确任务；提供上下文（文件路径、相关代码、报错）；指定输出格式；分步引导
- **AI 编程的局限性？** — 对复杂业务理解不足；容易产生有 bug 的代码；缺乏运行时理解

### 5.2 AI Coding Agent 架构

> 面试高频："你了解 Claude Code / Cursor 的源码吗？有特别了解的模块吗？"

- **AI Coding Agent 的核心架构？** — 本质是 Agent Loop：读取输入 → 组装上下文 → 调用 LLM → 解析响应 → 如果包含工具调用则执行 → 循环，直到模型输出纯文本
- **Agent Loop 关键组件？** — (1) System Prompt (2) 对话历史 (3) 工具注册表 (4) 工具执行器 (5) 终止条件
- **上下文管理？** — 接近上限时压缩早期对话为摘要；System Prompt 和最近几轮始终保留；工具结果可能被截断
- **子代理（Sub-agent）机制？** — 启动独立子代理处理隔离任务，避免污染主上下文；可并行执行；完成后只返回结论

### 5.3 Prompt Cache 与性能优化

- **Prompt Cache 是什么？** — 多次 API 调用的 prompt 前缀相同时，后续调用复用前缀的 KV Cache 计算结果，减少延迟和成本
- **TTL 是多少？** — 5 分钟。Agent Loop 中工具调用间隔通常在秒级，缓存命中率高
- **如何最大化缓存命中？** — 稳定内容放前面（System Prompt、工具定义）→ 变化内容放后面（最近对话）

## 6. AI 系统设计高频场景

### 6.1 客服 Agent 设计

- **整体架构？** — 用户请求 → 意图识别 → 路由 → 子 Agent/技能处理 → 回复生成
- **意图识别方案？** — 关键词匹配（简单）；分类模型（固定意图集）；LLM 零样本分类（开放意图）；混合
- **如何处理无法回答的问题？** — 转人工（置信度低于阈值）；多轮澄清；FAQ 兜底

### 6.2 知识库问答系统

- **如何设计企业知识库问答系统？** — 数据层：多源文档接入 → 解析 → Chunking → 向量化 → 索引；检索层：query 解析 → 混合检索 → 重排序 → 上下文组装；应用层：LLM 生成 → 校验 → 引用溯源
- **多跳推理怎么做？** — 迭代检索（逐步构建答案）；CoT 链式推理；图结构路径推理
- **如何评估 RAG 系统？** — 检索：Recall@K、MRR、NDCG；生成：RAGAS、faithfulness；人工评估

### 6.3 具身智能与多模态

- **具身智能（Embodied AI）？** — AI 通过身体（机器人、AR/VR）与物理环境交互
- **Computer Use？** — 让 Agent 通过屏幕截图理解 UI 并执行点击、输入等操作
- **多模态 Agent 的挑战？** — 视觉理解延迟；动作执行精确性；跨模态对齐；实时性要求
