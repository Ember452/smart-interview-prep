## 1. AI 辅助开发最佳实践

### 1.1 适合 AI 辅助的任务

- **样板代码生成**：CRUD 模板、DTO 转换、工具类
- **API 调用示例**：第三方 API（支付、短信、地图）的 SDK 使用
- **代码重构**：提取方法、拆分大类、命名优化
- **Bug 修复**：根据报错信息生成修复建议
- **测试用例生成**：单元测试、集成测试用例
- **设计模式应用**：将代码改写为设计模式实现

### 1.2 不适合 AI 辅助的任务

- **复杂业务逻辑设计**：需要深入理解业务规则和边界情况
- **系统架构决策**：技术选型、模块划分、依赖管理
- **性能优化**：需要实际压测数据支撑
- **安全性敏感代码**：认证授权、密码处理、加密解密
- **新技术/框架的第一次使用**：缺乏训练数据，容易出错

### 1.3 如何写出好的 AI 编程 Prompt

- 明确任务而非模糊描述
- 提供上下文（文件路径、相关代码、报错信息）
- 指定输出格式（注解、框架版本、代码风格）
- 分步骤引导而非一步到位
- 使用代码模板减少幻觉

## 2. AI 辅助代码审查

**使用流程：**
1. 提供完整上下文（代码 diff、相关测试结果）
2. 明确审查重点：性能、安全、可读性、异常处理
3. 分阶段审查：先看业务逻辑，再看技术实现

**AI 代码审查的优势：**
- 发现浅层问题：命名不规范、重复代码、缺少空检查
- 快速理解陌生代码
- 提供改进建议和示例代码

**AI 代码审查的局限：**
- 复杂业务逻辑仍需人工判断
- 无法发现运行时问题（空指针、并发问题）
- 对项目整体架构理解有限

## 3. 多级缓存架构追问链

> 当候选人提到"多级缓存"时，必须追问选型理由和场景适配。

### 3.1 本地缓存（进程内缓存）

**适用场景：**
- 数据访问频率极高（基础配置、热点数据）
- 对延迟极其敏感（<1ms 级别）
- 数据几乎不变
- 单机部署或可接受数据不一致

**不适用场景（应有反问）：**
- 大数据量 → JVM 堆内存浪费
- 多实例部署 → 数据不一致
- 频繁变更 → 缓存穿透问题

**常见实现：** Caffeine（推荐）、Guava Cache、ConcurrentHashMap + 定时刷新

### 3.2 Redis（分布式缓存）

**适用场景：**
- 需要跨进程/跨机器共享数据
- 数据有更新需要一致性
- 大数据量（不受单 JVM 堆内存限制）

**Redis 并不存在大量网络开销问题：**
- Redis QPS 可达 10万+，延迟可达亚毫秒级
- 本地缓存的主要目的不是"减少网络开销"，而是**极致的读取延迟**

### 3.3 L1 + L2 组合

```
请求 → 本地缓存(L1) → Redis(L2) → DB
命中 L1：延迟 < 1μs（纳秒级）
命中 L2：延迟 1-5ms
穿透 L1+L2：延迟 10-50ms
```

**什么时候需要 L1 + L2？**
- 极高频访问（每秒数万次）的热点数据
- Redis 集群仍然成为瓶颈
- 愿意为极致读取性能付出维护复杂度

### 3.4 追问链（必须连续追问）

1. **为什么要用本地缓存？** — 如果只说"减少网络开销" → 质疑：Redis 网络开销不是问题
2. **数据量多大？占多少内存？** — 如果数据量大 → 质疑：JVM 堆内存够用吗？
3. **多实例时本地缓存怎么同步？** — 如果没考虑 → 指出架构缺陷
4. **真的需要这么极致的性能吗？** — 如果没有充分理由 → 质疑是否过度设计

### 3.5 常见错误回答

| 错误回答 | 面试官质疑 |
|---------|----------|
| "减少网络开销" | Redis 网络开销很小，瓶颈真的在这吗？ |
| "数据量不大" | 那为什么不直接用 Redis？引入复杂度不值得 |
| "大家都这么用" | 社区常用不代表适合你的场景 |

## 4. 各技术栈 AI 辅助场景

### 4.1 后端（Java/Spring Boot）

- Service 层 CRUD 代码生成
- Controller 层 RESTful API 生成（含 @Valid 校验、@ControllerAdvice 异常处理）
- 单元测试生成（Mockito + JUnit 5）
- MyBatis XML 审查（N+1 查询、索引使用、全表扫描隐患）

### 4.2 后端（Go）

- HTTP handler 样板代码生成
- GORM 模型和查询方法生成
- goroutine 模式代码生成（worker pool、fan-in/fan-out）

### 4.3 前端（React/Vue）

- 组件样板代码生成（props 类型、状态管理）
- API 请求层封装（axios 拦截器、错误处理）
- 表单校验逻辑生成
- 样式代码（Tailwind/CSS Modules）

### 4.4 数据库

- SQL 查询优化建议（执行计划分析、索引推荐）
- JPA/Spring Data 命名查询方法
- 数据迁移脚本生成

### 4.5 API 设计

- RESTful API 设计辅助（遵循规范、路径命名、状态码）
- OpenAPI/Swagger 文档注释生成
- GraphQL schema 设计
