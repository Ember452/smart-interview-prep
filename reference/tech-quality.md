# 测试与质量知识库

> 面试高频问题与答案要点。覆盖测试分层/自动化/质量保障方向。

---

## 1. 测试分层

- **测试金字塔** — 比例：单元测试 70%（快速/低成本/大量）、集成测试 20%（检查组件间交互/边界）、E2E 10%（完整用户场景/高成本/易不稳定）。反金字塔（大量手工 E2E）是危险信号
- **单元测试设计原则** — 隔离外部依赖（Mock/Stub 替代 DB/HTTP）、快速反馈、测试行为不测试实现（即重构不破坏已有测试）、AAA（Arrange-Act-Assert 结构）
- **Mock vs Stub vs Fake** — Stub 返回预设值（不验证调用）；Mock 验证调用行为（预期被调用了哪些方法/参数/次数）；Fake 轻量级工作实现（如内存数据库替代真实 DB）用于集成测试

## 2. 自动化测试

### 2.1 框架选型
- **前端测试** — 单元：Jest/Vitest。组件：React Testing Library（按用户行为测试，不测内部 state）；Vue Test Utils。E2E：Playwright（多浏览器+自动等待+Tracing 录像）> Cypress（仅 Chrome 系+单 Tab 限制）
- **API 测试** — Postman/Newman CI 集成；REST Assured（Java DSL 风格）；SuperTest（JS 直接挂载 Express 测试）；Pact（契约测试，消费者端生成 contract JSON → 提供者端验证）

### 2.2 契约测试（Contract Testing）
- **Pact 工作流** — 消费者定义期望请求/响应 → 生成 pact JSON → 提供者加载 pact 文件并验证实际响应是否匹配。优势：微服务间接口变更早期发现，不依赖集成环境；避免联调阶段才发现接口不兼容
- **契约测试 vs 集成测试** — 契约测试：验证接口约定是否满足，不需启动真实依赖服务，速度快；集成测试：验证真实组件协作行为，需真实环境，成本高。二者互补不替代

### 2.3 Flaky Test 治理
- **常见 flaky 根因** — 时间依赖（固定 sleep 不准确→改用轮训等待条件）；执行顺序依赖（上一个测试改了共享状态→每个测试独立 set up/tear down，随机顺序执行）；环境不稳定（外部服务不可达→Mock/Stub 替代） ；异步未完成（UI 渲染未结束就断言→等待元素可见）
- **flaky test 管理** — 标记 flaky 从门禁中排除（但不能删除）；定期回访修复；按 flaky 率归类：连续失败 2+ 次→标记需修复

### 2.4 持续集成
- **CI 测试策略** — 预提交（lint + 受影响模块的单元测试 < 5min）→ 门禁（全量单元+集成 < 15min）→ 每日（E2E+性能/安全 可较长）。快速失败原则（先跑最快/最可能失败的测试）
- **覆盖率门禁** — 增量覆盖率（本次 MR 新增代码覆盖率≥80%）比全量覆盖率更有意义（全量有历史债）；业务核心模块覆盖率优先，不追求纯数字高的低质量覆盖

## 3. 质量保障

- **发布门禁** — 构建成功 + 单元/集成通过 + 增量覆盖率达标 + 安全扫描无高危漏洞 + 性能回退不超过阈值 + QA 冒烟通过。门禁由 CI pipeline 自动裁决，减少人工判断不一致
- **回归策略** — 风险驱动回归：优先回归高风险模块（历史 Bug 频发/核心业务/本次改动相关）。自动化回归（每次构建执行）+ 手工探索性测试（每次发布前 QA 1-2h 自由探索）。回归用例增量管理（每次上线 Bug 复盘 → 补充回归用例）
- **缺陷复盘（Blameless RCA）** — 不追责个人，聚焦系统为什么允许这个错误发生。时间线重建（什么时间/谁/什么操作/什么影响）→ 5-Why 根因分析 → 制定 Action Items（增加门禁/自动化验证/监控告警）→ 追踪至 Action Items 闭环
- **线上监控 × 测试联动** — 线上出 Bug → 复盘 RCA 提取测试缺口 → 补自动化回归用例 → 测试左移。线上异常指标（错误率/延迟突增）→自动触发回归测试套件→快速判定是否本次发布引入

## 4. 性能测试

- **测试类型** — 负载测试（逐步加压至目标 QPS，观察延迟/错误率是否达标）；压力测试（继续加压至系统崩溃，找到极限容量）；尖峰测试（突然从 0 加到峰值，验证弹性伸缩和降级策略）；浸泡测试（持续目标负载 24h+，发现内存泄漏/连接泄漏/磁盘满等长期问题）
- **工具选型** — JMeter（功能全面/插件多/学习曲线陡）；k6（JS 脚本/开发者友好/云原生）；wrk/hey（轻量 HTTP 压测）；Gatling（Scala DSL/高并发模拟能力强）
- **性能测试报告核心指标** — TPS（吞吐量/每秒事务数）、p50/p90/p99 延迟、错误率、CPU/内存/网络利用率。性能基准线建立后每次发布前回归，回退超过 5% 则阻断发布
