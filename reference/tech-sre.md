# 运维 / SRE 知识库

> 面试高频问题与答案要点。覆盖发布体系/故障管理/容量规划/自动化运维方向。

---

## 1. SLO/SLI/SLA 与可靠性工程

- **SLI/SLO/SLA 定义** — SLI（服务级别指标）：可量化的服务健康度指标（如可用性=成功请求数/总请求数）；SLO（服务级别目标）：基于 SLI 设定的目标值（如可用性 ≥99.9%）；SLA（服务级别协议）：与客户约定的正式承诺，违反 SLA 需赔偿（SLA 必须 ≤SLO，留余量）
- **Error Budget** — 错误预算 = 1 - SLO。99.9% SLO → 0.1% 错误预算（30 天允许宕机 43.2 分钟）。预算耗尽时：冻结功能发布，只做可靠性工作；预算充裕时：可激进发布/做实验。Error Budget 是 SRE 核心治理工具，将“可靠性”量化为可消费的预算
- **SLO 设计实践** — 用户视角选指标（不选 CPU/内存这类系统指标）；多 SLO 覆盖不同场景（可用性/延迟/数据新鲜度）；分级设定（P0 核心服务 99.99%、P1 重要服务 99.9%、P2 普通服务 99%）；定期 Review 调整（每季度复盘是否过严或过松）

## 2. 发布体系

- **灰度/金丝雀/蓝绿发布** — 灰度：按用户比例/地区逐步放量（5%→20%→50%→100%），每阶段观察指标无异常再继续。金丝雀（灰度子集）：部署少量新版本 Pod/实例处理部分流量，性能/错误对比旧版本，优于滚回成本低。蓝绿：维护两套完整环境（蓝=旧/绿=新），切换负载均衡一键整体切流，回滚瞬间（缺点：双倍资源）
- **回滚策略** — 按层级：代码回滚（git revert + 重新部署，最简单安全）优先于配置回滚 优先于 数据库 schema rollback（需前向兼容：先部署兼容新旧代码的中间版本 → 迁移数据 → 再清旧列）。触发条件：错误率/延迟超阈值自动触发回滚（无需人工决策，减少 MTTR）
- **变更管控** — 发布窗口（业务低峰期/周中非周五，保证有足够时间处理突发问题）、变更审批（生产变更需要 review/approval/记录变更理由和回滚方案）、封网期（大促/节假日禁非紧急变更）

## 3. 监控与告警

- **监控指标设计要求** — USE 方法（资源利用率 Utilization/饱和度 Saturation/错误 Errors）覆盖所有资源（CPU/内存/磁盘/网络）；RED 方法（Rate 请求速率/Errors 错误率/Duration 延迟）覆盖所有服务。Google 四金信号：延迟/流量/错误/饱和度
- **告警降噪全链路** — 告警规则合理阈值避免抖动（延迟 p99>2s 持续 5min 而非瞬时触发）→  分组（同一故障多条告警合并为一条，如 DB 慢→所有依赖服务的超时告警全归一组）→  抑制（底层基础设施故障抑制上层服务告警，只通知根因）→  静默（计划维护窗口/已知问题）→  值班升级（30min 未 Ack →电话；60min 未解决→ 管理层）
- **On-Call 质量度量** — 告警总量（每 on-call shift 告警数）、MTTA（平均确认时间）、MTTR（平均恢复时间）、告警误报率。目标：减少告警疲劳（一个 shift 被页超过 2-3 次是超标的）、减少非 actionable 告警（每条告警必须对应一个可执行操作）

## 4. 故障管理

- **故障响应流程** — 检测（告警触发/用户反馈）→ 声（Incident Commander 领导指挥调度/Operations Lead 技术排查/Comms Lead 沟通同步进度）→ 止损优先（切流量/回滚/限流/降级，不先找根因）→ 根因分析 → 恢复 → 事后复盘
- **Blameless 复盘（RCA）** — 5-Why：从故障现象逐层追问"为什么"找到系统层面根因而非个人操作失误。Action Items 必须是系统性改进（增加自动校验/门禁/监控/降级开关），不能只是"以后小心"。追踪 Action Items 至闭环
- **故障演练 / Chaos Engineering** — 故障演练：生产环境定期注入可控故障（如随机 kill Pod/网络延迟/依赖不可达），检验监控告警是否及时触发、值班响应效率、降级策略是否生效。不是破坏是主动验证韧性假设

## 5. 容量与成本

- **容量规划方法** — 历史趋势外推（基于过去 N 天指标增长预测未来资源需求）+ 业务增长因子（大促/新功能上线预估流量激增倍数）；非简单线性（增长到一定规模后资源需求可能跳变，如 DB 分片/缓存扩容）。压力测试验证新规格能否扛住
- **弹性伸缩策略** — 基于指标（CPU/memory/自定义 QPS）× 设定阈值触发；冷却时间（缩容后等待期，防止抖动频繁扩容回缩）。提前预热（预判大促流量提前扩容，流量高峰到达时已就绪）
- **成本优化方向** — 资源权利用满（requests 调至实际用量不浪费预留）、Spot/竞价实例（离线/非关键场景，价格节省 60-90%）、自动缩容（夜间/周末低峰缩到最小规格）、资源生命周期管理（自动回收/清理未使用资源/快照等）

---

## 6. 运维自动化

- **CMDB 设计** — 配置管理数据库是整个自动化运维的底座。核心数据：基础设施资源（实例/DB/KV/域名/负载均衡）与业务应用的关系映射（哪个应用依赖哪些资源）。数据准度是最大挑战（自动发现+同步 Agent 保证>人工录入更新）
- **ChatOps 实践** — 告警/变更/审批通知推送到 IM 群（飞书/钉钉/Slack），直接在 IM 中对话交互式机器人执行运维命令（重启服务/查询日志/创建发布单），所有操作记录可审计。降低运维门槛+提速响应
