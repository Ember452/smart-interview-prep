## 1. 渲染

### 1.1 渲染管线
- **图形渲染管线阶段** — 应用阶段（CPU 准备数据/Culling/合批/Draw Call 提交）→ 几何处理（GPU 顶点着色器/曲面细分/几何着色器/裁剪/屏幕映射）→ 光栅化（三角形遍历/生成片元）→ 像素处理（片元着色器/光照/纹理采样/透明混合/测试模板/深度写入）→ 输出到帧缓冲
- **Draw Call 优化** — 每次 Draw Call = CPU→GPU 通信开销（状态切换/数据提交），数量过多 CPU 瓶颈。优化：合批（Static Batching 合并静态不动物体/GPU Instancing 同网格同材质但矩阵不同）、合理设置材质复用减少 SetPass Call、SRP Batcher（URP/HDRP 兼容 Shader 自动加速）
- **前向渲染 vs 延迟渲染** — 前向：每个物体一遍渲染，一个 pass 中同时计算光照（光源数×物体数=复杂度，多光源慢，透明物体简单，MSAA 支持好）。延迟：先 GBuffer pass 写入几何信息（颜色/法线/深度），后光照 pass 根据 GBuffer 逐像素计算光照（光源数与物体数解耦，多光源友好，内存带宽大 GBuffer 占用高，透明物体/抗锯齿复杂）

### 1.2 Shader
- **顶点着色器 vs 片元着色器** — 顶点着色器：每个顶点执行一次（模型空间→齐次裁剪空间的空间变换/法线变换/UV 传递），输出顶点数据给光栅化器插值。片元着色器：每个像素片元执行一次（颜色计算/纹理采样/光照/Bump 映射/特效）。瓶颈通常在片元着色器（填充率/纹理带宽/复杂运算 per-pixel）
- **Shader 变体爆炸治理** — 不同宏组合/不同管线/不同光照模型生成海量 Shader 变体→编译时间长/运行时内存大。解决：剥离不必要宏、Shader Feature（避免 Multi_Compile 强制编译所有组合）、预编译预热（场景加载阶段完成所有变体编译，避免运行中触发造成卡顿）

## 2. 性能

### 2.1 帧率优化
- **Overdraw 过度绘制** — 多次绘制同一像素（UI 层叠/粒子叠加/透明物体）。优化：LOD（距离远→低面数模型）、Occlusion Culling（遮挡剔除，不可见物体不提交渲染）、Z-Prepass（先渲染深度→像素着色时深度测试先剔，减少无效的复杂着色计算）
- **纹理与内存带宽** — Mip Map（生成缩小版纹理链，远处物体自动用低分辨率纹理，降低带宽和锯齿）、纹理压缩（ASTC/ETC2 硬件直接解码压缩格式不用解压省带宽+内存，不能选未压缩大纹理如 RGBA32）、纹理 Atlas（多张小纹理合为一张 Atlas，减少纹理切换/ Draw Call 合并）

### 2.2 资源加载
- **异步加载与对象池** — 对象池（频繁创建销毁的对象预分配 N 个，用完激活而非销毁重分配，适合子弹/粒子/敌人）；Addressable/AssetBundle 异步加载（不阻塞主线程→加载完成后回调实例化）
- **场景切换优化** — 预加载下一场景资源（当前场景运行期间后台同时加载）、场景分块（Streaming 按玩家位置动态加载卸载周遭区块，避免一次性加载全地图）

### 2.3 内存管理
- **GC 与内存分配** — C# 堆内存（IL2CPP 下同样有 GC），减少每帧堆内存分配：避免字符串拼接/String.Format、缓存容器复用而非每帧 new、用 struct 替代小的频繁创建 class（struct 在栈上无 GC 压力）、对象池消灭频繁分配。Unity Profiler → GarbageCollector 定位每帧分配热点，GC Alloc 字段可精确到具体哪一行

## 3. 引擎

- **ECS/DOTS 设计思想** — Entity（ID 唯一，无数据无行为）→ Component（纯数据，无逻辑）→ System（纯逻辑，对拥有特定 Component 集合的 Entity 批量执行）。对比传统 OOP GameObject：数据与行为捆绑/多态/引用链长=缓存不友好。ECS 数据平铺线性排列=Cache 命中高/Jobs+Burst 编译成 SIMD 高性能多线程/自动批量并发处理海量实体
- **Unity vs Unreal 架构差异** — Unity：C# 脚本组件化架构（轻量/门槛低/灵活但大型项目架构易混乱）；Unreal：C++ 框架+Blueprint 可视化脚本（引擎源码开放/强框架约束 Gameplay Framework PlayerController/GameMode/Pawn 等角色分工明确，大型项目更易管理）。Unity 中大量逻辑写单体脚本 →Unreal 中同样功能需理解框架调用链

## 4. 网络同步

- **帧同步 vs 状态同步** — 帧同步 Lockstep：服务器只转发所有玩家操作帧序号→各客户端独立确定性模拟计算结果同步（一致性要求极高+浮点需定点数学+无延迟隐藏）→适合 RTS/MOBA 等低延迟局域网场景、高精度公平性。状态同步：服务器是唯一权威，计算结果同步到客户端（延迟补偿/差值平滑隐藏延迟/可服务器检测作弊）。多数 FPS/MMO 用状态同步
- **延迟补偿** — 服务端回退时间到玩家射击时刻，Rewind Replay 回滚碰撞检测，根据判定结果决定命中与否。加上客户端预测（客户端立即播放射击动画+先不等服务端确认=手感即时，服务端确认后矫正偏差）
- **断线重连** — 重连机制：客户端断网计时→尝试串断 N 次→超过阈值回主界面。服务端保留玩家状态一段时间（如 30s）等重连，超时踢出。重连后快速追状态（服务端一次性发送全量当前状，客户端在后台加速追帧缩小与服务器时间差→拉平后正常播放）

## 5. 游戏 AI 与物理

- **寻路算法** — A*（启发式搜索，F=G+H，障碍物复杂时效率高）+ Navigation Mesh（NavMesh 将可行走区域拆成凸多边形，路径点在多边形内插值平滑，避免贴墙走）。Unity 内置 NavMeshAgent，UE 内置 Navigation System
- **行为树（Behavior Tree）** — 节点类型：Sequence（顺序，全部子节点成功才成功）/ Selector（选择，任一子节点成功即成功）/ Decorator（装饰，条件判断/循环）/ Leaf（叶子，具体行为）。相比 FSM 更易扩展/复用/调试，中大型游戏 NPC AI 主流方案
- **物理引擎核心** — 刚体动力学（质量/惯性/重力/碰撞响应）；碰撞检测（粗筛：AABB 包围盒快速排除不可能碰撞对 → 精筛：GJK/SAT 算法精确判断是否相交 + 计算碰撞法线/深度）；约束求解器（关节/铰链/弹簧约束迭代求解）。PhysX（Unity/UE 默认）/ Havok（主机游戏/3A 大作）
- **游戏 AI 状态机** — FSM（有限状态机）：状态 + 转移条件（如血量<30% 进入逃跑状态），简单但状态多时转移爆炸。分层状态机 HFSM（子状态机嵌套，复用 Idle/Attack 等通用行为）。GOAP（目标导向行为规划）：给定目标和可用动作，AI 自己规划达成目标的动作序列，更灵活但计算成本高"
